# 인증 및 사용자 관리 구현 상세

## 개요
기본적인 로그인, 회원가입 기능과 더불어 세 가지 권한(관리자, 교수자, 학습자)에 따른 사이트 접근 및 유저 데이터 관리 구조를 구현했습니다.

## 주요 구현 사항

### 1. 인증 프로세스 (로그인 및 회원가입)
- **로직 처리**: `api.auth.login`, `api.auth.register`
- **검증**: `username`, `password` 일치 여부 확인 후 회원가입 시에는 중복 계정 가입 방지 체크 진행.
- **세션 유지 전략**: 현재는 세션 보안 테스트 목적으로 브라우저 `localStorage`에 `currentUser` 정보를 담아 유지 (차후 JWT 토큰 방식으로 교체 용이).
- **진입점**: `/` (로그인 폼 분리) 루트 페이지에서 권한(`currentUser.role`)을 판별하여 각각의 전용 대시보드(`/admin`, `/professor`, `/student`)로 즉시 리다이렉팅.

### 2. 권한 렌더링 방식 (Role-Based Access Control)
- 각 대시보드(`/admin`, `/professor`, `/student`) 페이지 최상단 컴포넌트 마운트 시 권한을 검증. 잘못된 권한 접근 시(예: 학생이 `/admin` 접근 시) 접근을 막고 로그인 페이지나 본인 대시보드로 반환.
- 네비게이션용 `Sidebar` 등 전역 컴포넌트 내부에서도 `currentUser.role` 값에 반응하여 조건부 UI(버튼, 탭 항목)를 능동적으로 출력.

### 3. 관리자(Admin) 전용 유저 관리 보드
- **경로**: `/admin` 대시보드의 '사용자 관리' 탭
- **제공 기능**: 전체 플랫폼 내 가입자 목록 조회 및 검색.
- **수정 권한**:
  - 아이디를 제외한 시스템 상의 전 사용자 이름/비밀번호 강제 수정 및 권한(Role) 롤백/승급 지원.
  - 가짜 계정이나 퇴사/자퇴자 계정에 대한 즉각적인 계정 삭제 조작 기능 포함.

## 4. [보완] 페이지 접근 제어 및 페이지 리다이렉션 흐름

LMS는 `useAuth` 훅의 `requireAuth` 함수를 통해 클라이언트 사이드에서 강력한 권한 검증을 수행합니다.

### 🔐 `requireAuth` 작동 메커니즘
각 대시보드 및 상세 관리 페이지의 `useEffect` 내에서 호출됩니다.
```javascript
const { requireAuth } = useAuth();
useEffect(() => {
    // 'admin' 권한이 없으면 자동으로 로그인 페이지 또는 이전 페이지로 튕김
    if (!requireAuth(['admin'])) return;
    loadData();
}, []);
```

### ↔️ 권한별 리다이렉션 흐름
로그인 성공 시 또는 루트(`/`) 접속 시 다음과 같은 흐름으로 자동 이동합니다.

1. **미인증 사용자**: `/` (로그인 폼) 유지
2. **인증 완료 사용자** (`currentUser.role` 기준):
    - `admin` ➔ `/admin` (시스템 통합 관리 보드)
    - `prof` ➔ `/professor` (강좌 및 수강생 관리 보드)
    - `student` ➔ `/student` (내 학습 공간 및 클래스 탐색)

### ⚙️ `ROLE_ROUTES` 설정 객체 기반 리다이렉션
하드코딩된 분기문 대신 유지보수성이 뛰어난 설정 객체를 사용합니다.
```javascript
const ROLE_ROUTES = {
    'admin': '/admin',
    'prof': '/professor',
    'student': '/student'
};
```
로그인 성공 시 `ROLE_ROUTES[user.role]` 값을 참조하여 즉시 해당 대시보드로 이동시키며, 새로운 역할이 추가될 경우 이 객체에 경로만 정의하면 시스템에 즉시 반영됩니다.

### 🛡️ 비인가 접근 시나리오
- 학생이 `/admin` URL을 직접 입력하여 접근 시: `requireAuth`가 `['admin']` 배열에 포함되지 않음을 감지 ➔ `showToast('접근 권한이 없습니다.')` ➔ `/student` 또는 `/`로 강제 리다이렉트.

67: ## 5. 상태 관리 무결성 및 방어적 프로그래밍 (State Integrity)
68: 
69: 서버 API의 응답이 불완전하거나 일시적인 오류가 발생하더라도 사용자 세션(ID, Role)이 파괴되지 않도록 하는 **무결성 보장 로직**이 적용되어 있습니다.
70: 
71: ### 🛡️ `AuthContext`의 상태 보존 메커니즘
72: `updateUser` 함수는 단순히 새로운 데이터를 덮어쓰는 것이 아니라, 기존 유저의 핵심 식별 정보를 보호합니다.
73: ```javascript
74: const updateUser = (updates) => {
75:     setUser(prevUser => ({
76:         ...prevUser,
77:         ...updates,
78:         // 업데이트 객체에 id나 role이 없거나 null이더라도 기존 값을 유지하여 세션 파괴 방지
79:         id: updates.id !== undefined ? updates.id : prevUser.id,
80:         role: updates.role !== undefined ? updates.role : prevUser.role
81:     }));
82: };
83: ```
84: 
85: ### 🚧 대시보드 진입 가드 (Hydration Guard)
86: 각 역할별 대시보드 페이지(`Admin`, `Professor`, `Student`)는 인증 정보가 완전히 복구될 때까지 비즈니스 로직 실행을 유예합니다.
87: - `useEffect` 내 `if (!user) return;` 체크를 통해 유저 객체가 유효할 때만 데이터 패칭을 시작합니다.
88: - 중요한 상태 변경 액션(클래스 생성, 수강 신청 등) 수행 직전에 현재 유저의 권한을 다시 한 번 검증하여 런타임 에러를 방지합니다.
